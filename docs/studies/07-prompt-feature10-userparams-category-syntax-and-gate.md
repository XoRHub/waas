# Fable 5 Prompt — Feature 10: `userParams` by category, removal of `expertUserParams`, and the missing `protocolParams` connect-time gate

Paste this document as-is as an implementation prompt. It assumes you (Fable 5) have no prior conversation context.

## Repo context

Feature 7 (already implemented and merged — commits `164b23a` CRD, `b9b7cde` api, `097e9b8` frontend, `3468bc4` dev catalog) introduced a two-list mechanism on `WorkspaceProtocol`: `UserParams []string` and `ExpertUserParams []string`, combined by union to form the allow-list of guacd parameter names overridable at connect-time — the distinction between the two lists being only **presentational** (`expert` vs `user` placement in the form, a name present in both rendering as `expert`).

This prompt **changes the doctrine** on three points decided since then, and invalidates part of Feature 7's design. This is not an addition: it's a replacement.

## What already exists (know this before coding)

### The `Category` registry (`operator/pkg/params/params.go:44-69`) — DOES NOT CHANGE

`Category` has 7 values (`display`, `audio`, `input`, `clipboard`, `session`, `security`, `connection`), assigned to every `Param` in the table (`params.go:109-448`), and `AllCategories()` (line 64) gives the canonical order. `ForProtocol(protocol)` (line 454) filters by protocol and sorts by category then tier then name. **This mechanism stays as-is** — it's the building block this prompt reuses for the new `userParams` syntax, not something to redo.

### `UserParams`/`ExpertUserParams` — TO BE REMOVED ENTIRELY

CRD field (`operator/api/v1alpha1/workspacetemplate_types.go:235-249`):
```go
UserParams []string `json:"userParams,omitempty"`
ExpertUserParams []string `json:"expertUserParams,omitempty"`
```
`ExpertUserParams` no longer has a reason to exist: Feature 7 justified it by a placement need (expert vs user in the form), but that placement is already and solely driven by `Tier` (`params.go:23-42`, `ui`/`advanced`/`platform`) — `ExpertUserParams` duplicated information the registry already carries. Remove the field and **every reference** to `ExpertUserParams`/`expertUserParams` in the repo — the exhaustive search as of this writing (`grep -rn "ExpertUserParams\|expertUserParams"`) touches:

### The `protocolParams` gate — CONFIRMED VALIDATION HOLE, TO BE CLOSED

At connect-time, `workspace_service.go:539-568`:
```go
isAdmin := actor.Role == string(auth.RoleAdmin)
allowList := params.UnionNames(entry.UserParams, entry.ExpertUserParams)
if violation := params.ValidateUserOverrides(protocol, in.Params, allowList, isAdmin); violation != nil {
    return nil, apierror.Forbidden(violation.Error())
}
if len(in.Params) > 0 && !isAdmin {
    if pol := s.actorPolicy(ctx, actor); pol != nil && pol.Spec.Overrides != nil &&
        !slices.Contains(pol.Spec.Overrides.AllowedFields, waasv1alpha1.FieldProtocolParams) {
        return nil, apierror.Forbidden(...)
    }
}
```
This code ONLY checks the right on the **`WorkspacePolicy`** side (`pol.Spec.Overrides.AllowedFields`). It NEVER checks `tpl.Spec.Overrides.AllowedFields` (the template's own `TemplateOverrides`, `workspacetemplate_types.go:270-275`, exposed via `tpl.Spec.FieldOverridable(field)` — a method already written, `workspacetemplate_types.go:485-497`, `if s.Overrides == nil { return false }`).

Compare with `policy.CheckOverrides` (`operator/pkg/policy/policy.go:552-574`), which is the reference for overrides at workspace CREATION time: it requires the field in **both lists** (`checkField`, lines 564-574) — the template's AND the policy's — except for admins (full bypass) or the template's owner (bypass only on the template side, still subject to the policy). Connect-time today only applies half of this rule, and also has no notion of owner. §B closes both gaps.

**Concrete case reproducing the bug today**: `hack/dev/templates-dev.yaml:151-158`, the `ubuntu-firefox` template — `userParams: [enable-audio, color-depth]` but **no `overrides` block at all** (so `FieldOverridable` returns `false` for every field, including `protocolParams` — this "disposable session" template is deliberately locked down). With the current code, a non-admin user connecting to a workspace based on this template and sending `color-depth` as a connection param sees the request accepted: `ValidateUserOverrides` validates it (the name is in `userParams`) and there's no policy assigned to the actor to block it (or if the policy allows `protocolParams`, nothing on the template side objects). This is the hole: the template **never explicitly authorized** any connect-time tweak, and yet `userParams` produces an effect.

## What needs to be delivered

### A. `userParams`: entries by name OR by category

Each element of the `UserParams` list (the `ExpertUserParams` field disappears, there's only ONE list left) can be:
- an **exact parameter name** (current behavior, unchanged — backward compatible with all existing templates that only use raw names);
- a **category selector**, prefixed `cat:`, e.g. `cat:audio` — authorizes ALL parameters of that category for that protocol, resolved dynamically against the `params.go` registry at validation time (webhook AND api-server), including parameters added to that category later without touching the CRD or the template. This is explicitly the intended goal: not depending on a CRD or template change when a new parameter appears in an already-delegated category.

**Decided — no third combined form** like `cat:audio;params:enable-audio,audio-servername` in a single string: it brings nothing over the two forms above. `["cat:audio"]` alone already delegates all of `enable-audio` + `audio-servername` (and any future `audio` parameter); to carve out a precise subset without following the category's future evolutions, two raw entries (`["enable-audio", "audio-servername"]`) already do it without new syntax. A mini-language with internal separators (`;` then `,`/`/`) would have complicated parsing for a use case that doesn't exist.

**Decided — semantics in case of coexistence of `cat:X` + an individual name from that same category in the same list: no conflict, no priority.** Every entry is purely additive (it grants, never removes): the union of `cat:audio` + `audio-servername` is strictly identical to `cat:audio` alone, the individual name's presence is redundant but not contradictory — no "unit > category" priority rule is needed or meaningful here, that would only arise if an exclusion syntax were introduced (out of scope, cf. above). Treat this case in resolution as a simple deduplicated duplicate (`ResolveUserParamNames`, see below, must not error or have any special behavior on it).

Wire the category → names resolution everywhere `UserParams` is today consumed as a flat list of names:
- **Webhook** (`workspacetemplate_webhook.go:66`, `ValidateUserParamNames`): a `cat:X` entry must be recognized as valid if `X` is a known `Category` (`params.go` `AllCategories()`), and doesn't need to literally exist as a param name — don't make it fail by treating it as an unknown param name.
- **api-server** (`template_service.go:253`, same function): same.
- **api-server connect-time** (`workspace_service.go:553-554`, `ValidateUserOverrides`): the effective allow-list passed to the function must be the full expansion — each `cat:X` replaced by every name from `params.ForProtocol(protocol)` whose `Category == X` (excluding `TierPlatform`, which stays banned as today), merged (deduplicated) with the raw names. **Decided — new dedicated function** in `params.go`, e.g. `ResolveUserParamNames(protocol string, entries []string) []string` (name is your choice, but new — don't reuse `UnionNames`, removed, cf. above) rather than duplicating this expansion in every caller.

**Decided — where the `cat:X` → names resolution lives (server architecture = single source of truth):**
- **Template editor** (`template_service.go` DTOs → `TemplatesPage.tsx`): exposes and consumes the **raw** list (`cat:X` intact) — this is the configuration itself being edited, and the UI (see below) needs to know whether a category is delegated AS A WHOLE (`cat:X` literally present) or only partially.
- **Connect-time exposure** (`workspace_service.go:929-932`, the model served to `ConnectionSettingsDialog.tsx`/`CreateWorkspaceDialog.tsx`/`SessionOverlay.tsx`): exposes the list **already resolved** by `ResolveUserParamNames` — these forms have never needed to know about the notion of category to build their allow-list, only a flat list of names (as today with `unionNames`). The frontend therefore does NOT reimplement the `cat:` expansion: it receives concrete data, no syntax to parse, which stays consistent with the "registry = single source of truth" architecture already established in Feature 7.

**Template editor (`TemplatesPage.tsx:489-528`)**: the current `locked/user/expert` tri-state (Feature 7) becomes a **binary `locked/user`** per parameter (no more `expert`), plus a per-section/category toggle action ("allow the whole Audio category" → adds `cat:audio` to `userParams`, removes individual names from that category that would already be there to avoid redundancy, cf. dedup above). **Decided — rendering**: if `cat:X` is literally present in the raw list, the category displays as "full" (delegated as a whole, e.g. a section header marked "authorized"); otherwise, the section stays visible with each parameter individually toggleable, and any parameter not named in the list appears grayed-out/disabled rather than hidden (the admin sees what they could delegate, not only what's already delegated). No need to detect by counting the edge case "all names of a category checked one by one without `cat:X`": that still displays visually as "each box checked", not "category full" — not a problem to solve, just a slightly different rendering of a functionally equivalent case.

### B. The template `protocolParams` gate at connect-time (with owner bypass)

In `workspace_service.go`, before or alongside the existing policy check (lines 561-567), add the symmetric check on the template side, **with the owner bypass mirrored from `policy.CheckOverrides`**:
```go
if len(in.Params) > 0 && !isAdmin {
    isOwner := tpl.Spec.Overrides != nil && tpl.Spec.Overrides.Owner != "" && tpl.Spec.Overrides.Owner == actorUsername
    if !isOwner && !tpl.Spec.FieldOverridable(waasv1alpha1.FieldProtocolParams) {
        return nil, apierror.Forbidden(fmt.Sprintf(
            "template %q does not allow overriding %q (allowed: %v)", tpl.Name, waasv1alpha1.FieldProtocolParams, tpl.Spec.Overrides))
    }
    // ... existing policy check, still applied even to the owner ...
}
```
Without the template right (or owner status), `userParams` (regardless of its syntax, §A) must produce NO effect at all — even if a name or category is listed there, the non-admin/non-owner user cannot tweak any param at connect-time as long as the template's `overrides.allowedFields` does not contain `protocolParams`. Verify with the `ubuntu-firefox` test case (§ above, no `overrides` block) that this case is now rejected for an ordinary user.

**Decided — owner bypass: YES, mirror `policy.CheckOverrides`.** The field's comment (`workspacetemplate_types.go:277-279`) is explicit: *"Owner is the platform username owning this template: that user may override any field on workspaces stamped from it, **like an admin**"* — this isn't a concept scoped to workspace creation, it's an "admin for this template" right that must therefore apply symmetrically at connect-time. The owner bypasses the **template** gate but remains subject to the **policy** gate (identical to `CheckOverrides`, line 570-572 — `policyAllows` always applies). Retrieve `actorUsername` using the pattern already in place in this same file (`workspace_service.go:616-624`/`634-642`: `s.users.FindByID(ctx, actor.ID)` → `.Username`), don't reinvent another path to get the actor's identity.

- Regenerate the CRD (`operator/config/crd/bases/...yaml`, `helm/waas/crds/...yaml`, `zz_generated.deepcopy.go`) via the `manifests` target of `operator/`'s Makefile, don't edit them by hand.
- Regenerate `docs/guacd-parameters.md` (`make docs-params`) if its generation references `expertUserParams` or the new `cat:` syntax.
- Dedicated Go test for: (1) the `cat:X` → names resolution (nominal case, unknown category, a category containing only `TierPlatform` for a given protocol → empty list rather than error, up to you to decide and document, coexistence of `cat:X` + an individual name from the same category → result identical to `cat:X` alone), (2) connect-time rejection when the template's `overrides.allowedFields` lacks `protocolParams` for a non-owner/non-admin actor (reproduce the `ubuntu-firefox` case), (3) connect-time acceptance for the template's owner despite the absence of `protocolParams` in the template's `allowedFields`, but rejection if that owner's policy doesn't allow it either, (4) update `connect_params_test.go` to remove the `ExpertUserParams` cases and replace them with `cat:` cases.

- Vitest test: the template editor (§A) consumes the raw list with "category full" vs "partially grayed out" rendering, the connection form consumes the already-resolved list exposed by the api-server (no `cat:` expansion on the frontend side) — without regression on the existing `ParamField.test.ts`/`ProtocolParamsForm.test.tsx` tests (adapt those that explicitly tested the `userParams`/`expertUserParams` overlap).
