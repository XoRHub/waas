# Audit WaaS — rapport complet (2026-07-08)

Audit composant par composant (`operator/`, `api-server/`, `frontend/`,
`wwt/`, `hack/`), sans implémentation. Chaque constat est sourcé
(fichier + ligne) ; les chiffres de couverture sont mesurés, pas
estimés. Corrections de périmètre par rapport au brief : l'opérateur
est en **controller-runtime Go** (pas Kopf), le backend s'appelle
`api-server/`, et le CI canonique est **GitHub Actions** (GitLab en
transition, retrait planifié).

## 2. Doublons — avec les divergences qui ont déjà mordu

### D1 (le plus important) — la liste des protocoles existe en 4 exemplaires

- `operator/api/v1alpha1/workspacetemplate_types.go:207` (enum kubebuilder)
- `operator/api/v1alpha1/workspaceimage_types.go:9` (enum kubebuilder, 2ᵉ copie)
- `operator/pkg/params/params.go:441` (`Protocols()`)
- `api-server/internal/service/governance_service.go:331` (switch `AdminUpsertImage`)

**Preuve de nocivité vécue** : l'ajout de kasmvnc a exigé 4 retouches
manuelles, et le switch (4ᵉ copie) a failli être oublié. Aucun test
d'exhaustivité ne relie ces 4 sources — contrairement au registre des
overridable fields qui a exactement ce garde-fou
(`operator/pkg/policy/overrides_registry_test.go`, modèle à répliquer).
*Factorisation : `Protocols()` = source unique + test « enum CRD ==
Protocols() » (scan YAML, comme le fait déjà le test des overridable
fields) + switch remplacé par un lookup.*

### D2 — Types TS écrits à la main vs modèles Go

`frontend/src/types.ts` (642 lignes) recopie
`api-server/internal/model/model.go`. Divergence déjà payée : la
régression `params: null` (le type TS mentait sur la nullabilité).
Mensonges latents du même genre : tout champ Go slice/map **sans**
`omitempty` sérialise `null` là où TS déclare un tableau.
*Factorisation : génération (tygo, ou OpenAPI + openapi-typescript —
`docs/openapi-governance.yaml` existe mais manuel et partiel, donc
lui-même un doublon qui dérive).*

### D3 — Phases dupliquées

Enum Go (`operator/api/v1alpha1/workspace_types.go:13-25`) vs littéraux
TS éparpillés (`frontend/src/lib/lifecycle.ts:18-23`,
`frontend/src/pages/PortalPage.tsx:448,493,496`,
`frontend/src/lib/target.ts`). Pas d'incident encore ; une phase
renommée côté Go casserait silencieusement les cartes. *Union type TS
unique dans types.ts (ou D2 le résout).*

### D4 — Copie-de-secret-vers-ns-du-pod ×2

`operator/internal/controller/kasm_credentials.go` (~60 l.,
per-workspace, owner-ref/teardown) et
`operator/internal/controller/pull_secret.go` (~70 l., partagé, label
dédié) implémentent deux fois « ensure Secret dans le ns cible avec
convergence ». Sémantiques volontairement différentes mais squelette
get/create/update-on-drift identique. *Factorisation possible en helper
`ensureSecretCopy(key, data, labels, owner)` — gain modeste, à faire à
la 3ᵉ occurrence.*

### D5 — micro-doublons (cosmétique)

`envOr` ×2 (`wwt/cmd/main.go:61`,
`api-server/internal/config/config.go:182`) ; la const CSS
`field = 'mt-1 w-full rounded-md…'` répétée dans ≥4 dialogs ;
`eventLastSeen` (les 3 timestamps d'Event) restera unique tant qu'un 2ᵉ
consommateur n'apparaît pas.

**Duplication sémantique surveillée mais SAINE** : la règle « ce que le
portail affiche = ce que le webhook décide » n'est PAS dupliquée — les
deux appellent `pkg/policy` (meilleure décision d'architecture du repo,
à protéger).

## 3. Standards par composant

### Operator — globalement conforme, 3 écarts importants

✅ Conformes (vérifiés) : status subresource + `metav1.Condition` +
`observedGeneration`, printcolumns, events sur transitions, finalizer
idempotent, RBAC énuméré **sans wildcard** + garde `rbac_test.go`
(generated ⊆ helm), leader election + probes câblés dans le chart
(`helm/waas/templates/operator.yaml:150,178`), reads Secrets/ConfigMaps
hors cache, pas d'état mémoire entre reconciles (`Now`/`Probe`
injectables = configuration, pas état).

| Gravité | Constat |
|---|---|
| **Important** | **Pas d'envtest** : 70,6 % de couverture contrôleur mais 100 % fake-client. Defaulting CRD, **CEL** (le one-of image/registry !), SSA/conflits, GC réel ne sont exercés que par le smoke k3d. Le fake client a déjà masqué un bug réel (pvc-protection vs cleanupNamespace, cf. `docs/workspace-deletion.md`) |
| **Important** | `ensureDeployment/STS` **create-only sur le podTemplate** (`operator/internal/controller/workload.go:205+`) : un template édité (image, env, resources) ne converge jamais vers les workloads existants — sauf le hash kasmvnc, seul champ convergé. Choix implicite devenu règle : à **documenter comme décision** (ou converger champ par champ) — c'est la 1ʳᵉ surprise d'un admin |
| **Important** | CRDs `v1alpha1` sans stratégie d'évolution écrite (pas de plan conversion/v1beta1, champs ajoutés en continu). OK aujourd'hui, à formaliser avant tout utilisateur externe |
| Cosmétique | Labels `app.kubernetes.io/name`/`instance` absents des objets créés par l'opérateur (seul `managed-by` + `waas.xorhub.io/*`) ; `workspace_controller.go` = 921 lignes (découpage lifecycle/status/finalize possible) |

### api-server (Go) — idiomatique, 2 écarts importants

✅ : erreurs wrappées systématiquement, `context` propagé, problem+json
uniforme, validation d'entrée server-side réelle, autorisation 100 %
serveur (`fetchByID`, actor), secrets jamais loggés (vérifié sur les
chemins credentials), goroutines à ctx (sweepers, watches), golangci v2
configuré, DI par constructeurs testable.

| Gravité | Constat |
|---|---|
| **Important** | `http.Server` avec **seul `ReadHeaderTimeout`** (`api-server/cmd/api-server/main.go:143-146`) : pas de `ReadTimeout`/`IdleTimeout`. Nuance : `WriteTimeout` doit RESTER 0 (il tuerait le SSE) — mais ça mérite le commentaire, et les deux autres manquent |
| **Important** | Couche SQL **dual-backend** (pgx prod / sqlite dev) à **15,5 %** de couverture, 2 fonctions de test : les subtilités déjà rencontrées (scanners RFC3339, `repository/time.go`) prouvent que c'est une zone à bugs de divergence |
| Cosmétique | `governance_service.go` 811 l. / `workspace_service.go` 914 l. — services fourre-tout, découpables |

### Frontend — hygiène excellente, filet de sécurité quasi absent

✅ : `strict: true` effectif, **0 `any`**, **0 `console.log`**,
react-query cohérent partout (un seul pattern d'état serveur),
loading/error/empty géré dans les composants récents, aucune logique
d'autorisation côté client (tout est re-fetch autorisé).

| Gravité | Constat |
|---|---|
| **Important** | **Aucun eslint/prettier** (`tsc -b` est le seul lint). Pas de règles react-hooks/exhaustive-deps : les effets de `DesktopPane.tsx` (l. 144-299) dépendent d'une discipline manuelle |

| **Important** | Nullabilité : le contrat `?? []` est appliqué au cas par cas ; sans D2 (génération de types), chaque nouveau champ Go re-crée le risque `params: null` |
| Cosmétique | PortalPage monolithe (dialogs à extraire) ; a11y de base non auditée (boutons icônes sans aria-label dans SessionCard/menus) |

### wwt — bon état

✅ : 87 / 86,8 / 62,2 % de couverture sur guac/kasm/proxy, invariant de

## 5. Dette technique & plan d'action priorisé

**Dette inventoriée** (le repo n'a aucun TODO — la dette vit dans les
docs, ce qui est mieux que la moyenne) :

- N+1 documenté non corrigé (`docs/diagnostics/workspace-list-loading.md`
  — Get template par workspace × poll quota 15 s ; la solution est déjà
  écrite dans le diagnostic).
- Clipboard non gouverné sur le chemin kasmvnc (décision v1 assumée,
  phase 4 kasm = DLP dérivé de la policy).
- `guacamole-common-js ^1.5` vs guacd 1.6.0 (`frontend/package.json:15`
  vs `helm/waas/values.yaml:141` — compatible mais à aligner).
- OIDC jamais testé contre un IdP réel.
- Refs `registry.xorhub.io` dans gitops (4 occurrences).
- Pin toolchain golangci local vs CI (incident documenté).
- `.gitlab/` + `cliff.toml` à retirer à la fin de la transition GitHub.

**Plan (quick wins → chantiers)** :

| Ordre | Chantier | Problème / bénéfice | Risque | Effort |
|---|---|---|---|---|
| 1 (QW) | Garde d'exhaustivité protocoles (D1) | tue la classe de bug la plus récemment vécue | nul | ~1 h |
| 2 (QW) | eslint + prettier + react-hooks + job CI | filet frontend inexistant → règles hooks/a11y | faible (hygiène déjà bonne) | ~½ j |

| 5 | Tests handlers + repository dual-backend (#1, #2) | zones les plus risquées du backend | faible | ~2-3 j |

| 9 | Fix N+1 documenté | perf portail | faible | ~½ j |

**Anti-dette (empêcher le retour)** :

- (a) Job CI eslint + prettier.
- (b) Génération de types en CI avec `git diff --exit-code` (même
  pattern que le drift check CRD existant — mécanisme déjà culturel
  dans ce repo).
- (c) Étendre le pattern « registre + test d'exhaustivité »
  (overridable fields) aux protocoles et aux phases.
- (d) Ratchet de couverture par paquet sur les zones remontées
  (handler ≥ 40 %, repository ≥ 50 %).
- (e) Pre-commit léger gofmt + eslint (optionnel — la CI est déjà
  bloquante).
- (f) Convention documentée + lint custom : pas de `json:"…"` sans
  `omitempty` sur les slices/maps des modèles API (LA racine de
  `params: null`).
