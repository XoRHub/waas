# ADR 0002 — CRD evolution strategy

**Status**: accepted (2026-07-08), "classic" strategy.

1. **While `v1alpha1`**: **additive-only** changes —
   optional fields, extend-only enums (guarded by the
   exhaustiveness tests: protocols, overridable fields). Never a
   rename/retype in place. Any breaking need ⇒ a new version.
2. **Move to `v1beta1`**: triggered by the first external user
   OR the first breaking need. Mechanics: conversion
   webhook (controller-runtime), double-serve `v1alpha1`+`v1beta1`
   for at least one release, then switch the storage version.
   **Prerequisite: the envtest suite** (T10 of the post-audit plan) — an
   untested conversion against a real apiserver is not a
   conversion.
3. **Deprecation**: `// +deprecated` marker on the field + release-notes
   mention; removal at the earliest two releases later.
4. **Existing mechanics reused**: CRDs generated by
   `make manifests` and drift-checked in CI; release-please carries
   the CRD change announcement in the release notes.
