# KasmVNC — data plane, exposed API surface, configuration, licensing

WaaS serves the official Kasm workspace images (`docker.io/kasmweb/*`)
**without the Kasm Workspaces platform** — WaaS stays the broker. These
images bundle KasmVNC, which broke with RFB and is browser-only (HTTPS
WebSocket on port 6901, self-signed cert, HTTP Basic auth): **guacd
cannot connect to it**, so `kasmvnc` is a separate data plane, exclusive
on a template (see `docs/templates-and-protocols.md`).

## Data plane

The browser renders KasmVNC's own web client in a same-origin iframe;
`wwt/internal/kasm/kasm.go` reverse-proxies everything under
`/kasm/{sessionID}/…` (HTML, assets, the `/websockify` WebSocket) to
the pod's 6901 endpoint. Unlike the guacd path — one WebSocket per
session — the whole client app flows through this handler.

- **Auth**: nothing is proxied before the platform connection JWT is
  validated. The first request carries `?token=…` (the iframe URL); the
  handler answers with a session-scoped cookie whose value IS the JWT
  (stateless, same expiry). The cookie never goes upstream — the
  `Cookie` header is stripped.
- **Upstream credentials**: the per-workspace `VNC_PW` Secret
  (`waas-kasm-<name>`, generated by the operator) is injected
  server-side as `Authorization: Basic` for user `kasm_user`
  (`kasm.go`, `proxyTo`) — it never reaches the browser. The proxy also
  sets an `Origin` header when absent: KasmVNC's WebSocket handshake
  parser requires one (browsers always send it, health probes don't).
- **Session end**: `httputil.ReverseProxy` only returns once the
  upgraded WebSocket closes — that return is the session-end signal
  (`EndSession`), with the api-server's SessionSweeper as safety net.
- **Remote workspaces**: kasmvnc is refused there (400) — the kasm
  proxy targets an in-cluster pod, "external machine" semantics has no
  kasm equivalent (`docs/remote-workspaces.md`).

### Upstream TLS posture

The kasmweb images ship `network.ssl.require_ssl: true` with a
**self-signed certificate**, so the proxy's transport skips
verification (`InsecureSkipVerify`, `kasm.go`). The hop is pod-to-pod
inside the cluster and netpol-guarded. Replacing the self-signed cert
with a cert-manager-issued one is a known pending hardening item
(originally "phase 4" of the kasm feasibility plan) — not implemented.

## Exposed `/api` surface

Standalone KasmVNC serves an HTTP API under `/api/…` — nine endpoints,
all requiring owner credentials. `kasm_user` **holds the owner role**
on its own session (`.kasmpasswd` carries the `wo` — write + owner —
flags; verified live 2026-07-10), and the proxy injects the Basic pair
on every request, so a WaaS user's browser reaches this API through the
proxy.

| Endpoint | Effect | Proxied? |
|---|---|---|
| `/api/downloads` | Lists + serves files from the session's downloads folder | **Blocked (403)** |
| `/api/clear_clipboard` | Clears the KasmVNC clipboard and the session's X clipboard | Open |
| `/api/get_screenshot` | Screenshot of the current session | Open |
| `/api/create_user`, `/api/update_user`, `/api/remove_user` | Manage the session's user accounts | Open |
| `/api/send_full_frame` | Forces a full frame resend to read-only users | Open |
| `/api/get_bottleneck_stats`, `/api/get_frame_stats` | Session telemetry | Open |

- **`/api/downloads` is blocked** at the proxy (`isDownloadsAPIPath`,
  `kasm.go`) because it is the same session→browser file egress the
  guacd path gates behind `TierPlatform` for
  `enable-sftp`/`enable-drive` ("until the file-transfer feature ships
  with its own policy gate", `operator/pkg/params/params.go`). No such
  gate exists for kasmvnc yet, so the endpoint is blocked outright to
  keep file egress governed identically across protocols. The path is
  normalized and lowercased before matching so trailing-slash, dot and
  double-slash variants can't slip past.
- **The other eight stay open by documented choice**: a WaaS kasmvnc
  session is single-user and self-owned, with no third party to protect
  the user from — they already have the keyboard and screen.
  `get_screenshot` only leaks their own desktop to their own browser;
  `create_user`/`remove_user` only create accounts on a pod for which
  wwt remains the sole entry point, injecting `kasm_user` regardless.
  Blocking them would be defending against the user's own capabilities
  on their own session.

## Capabilities vs the Kasm platform

Verified against upstream docs and maintainer-settled issues
(2026-07-10):

- **Standalone KasmVNC has**: clipboard, dynamic resize, multi-monitor
  (native "Display Manager"), the downloads API above.
- **Audio (playback + mic)**: absent from standalone KasmVNC at every
  version — a Kasm Workspaces (commercial platform) exclusive
  ([kasmtech/KasmVNC#31](https://github.com/kasmtech/KasmVNC/issues/31)).
- **Upload (browser → session)**: absent at every scale and not planned
  upstream ([kasmtech/KasmVNC#209](https://github.com/kasmtech/KasmVNC/issues/209)).

## Configuration: file hierarchy and the WaaS merge

KasmVNC itself deep-merges three files, key by key, lowest to highest
priority (official reference:
<https://kasmweb.com/kasmvnc/docs/latest/configuration.html>):

1. `/usr/share/kasmvnc/kasmvnc_defaults.yaml` — the full defaults
   shipped by the image (resolution, encoding, SSL, brute-force,
   default DLP block…).
2. `/etc/kasmvnc/kasmvnc.yaml` — system override.
3. `~/.vnc/kasmvnc.yaml` — user override, **where WaaS mounts its
   per-workspace ConfigMap** (single-file subPath; the `.vnc` directory
   itself is an operator-managed emptyDir so it stays writable for
   KasmVNC's runtime artifacts — `self.pem` lives there — whatever uid
   the image runs as).

Because the image's defaults already apply to every key the WaaS file
leaves unspecified (verified live: a partial user file inherits
resolution, encoding, ssl… from `kasmvnc_defaults.yaml`), **WaaS adds
no base/defaults layer of its own** — re-baking those defaults into a
WaaS constant would freeze them against future image versions. What
WaaS writes into `~/.vnc/kasmvnc.yaml` is TWO layers merged
(`operator/internal/controller/kasm_config.go`):

1. the admin's opaque `WorkspaceTemplate.spec.kasmvncConfig`
   (convenience config, every upstream option works), then
2. the clipboard DLP keys derived from `WorkspacePolicy.Clipboard`
   (the security authority), stamped last.

A content change rolls the workload: subPath mounts never refresh in
place, so the content hash is stamped on the pod template
(`annotationKasmConfigHash`).

`kasmvncConfig` is editable from the admin template editor (shown only
when a `kasmvnc` protocol is present, same guard as the webhook), and
the effective merged file is readable by the workspace owner via
`GET /api/v1/workspaces/{id}/kasmvnc-config`.

### Clipboard enforcement (policy-derived DLP)

Denying clipboard in the policy actually disables copy/paste inside the
container — not just a greyed-out button. At reconcile, the operator
resolves the workspace **owner's** policy (container-level DLP can only
enforce one policy per workload; on personal kasmvnc namespaces owner
== user; resolution failure fails closed) and stamps three keys, the
single source being `operator/pkg/kasmcfg.ManagedClipboardKeyPaths`:

```yaml
data_loss_prevention.clipboard.server_to_client.enabled: <copyAllowed>
data_loss_prevention.clipboard.client_to_server.enabled: <pasteAllowed>
runtime_configuration.allow_client_to_override_kasm_server_settings: false
```

The third key is always forced off — without it the KasmVNC client can
re-enable the clipboard at connect time (the official docs are explicit
about this). Because the controller overwrites these keys, a template
that sets any of them by hand would be silently ignored — so both the
admission webhook (`workspacetemplate_webhook.go`) and the api-server
template validation (`template_service.go`, mirrored as a clean 400)
reject them, pointing the admin at `WorkspacePolicy.Clipboard`. The
unmanaged clipboard sub-keys (`size`, `allow_mimetypes`,
`delay_between_operations`, `primary_clipboard_enabled`) remain the
admin's. Full precedence map across both data planes:
`docs/clipboard.md`.

## Images, placement, licensing

- Catalog entries for `docker.io/kasmweb/*` are approved by **fixed
  release tag** (`tagPolicy: tag`, the bootstrap default in
  `helm/waas/values.yaml`): upstream publishes versioned tags
  (`1.19.0`) and `catalog-kasmweb.yaml` references them by tag. A
  digest requirement was tried first and dropped — it denied every
  entry of the synced catalog (which carries tags, not digests) for no
  real gain; admins who want immutability can still pin `@sha256:…`
  per entry or set `tagPolicy: digest` themselves. Some images are
  multi-arch
  (`core-ubuntu-*`, `ubuntu-noble-desktop`, `firefox`, `vs-code`,
  `terminal`), some amd64-only (`desktop`, `chrome`) — the existing
  `Architectures` → nodeAffinity placement covers this.
- Home lives at `/home/kasm-user` (template `homeMountPath`); `KASMVNC_AUTO_RECOVER`
  brings Xvnc and the app back after a pod restart. A Memory emptyDir on
  `/dev/shm` (via the `volumes` passthrough) is recommended — 512Mi is
  enough for Firefox.
- **KasmVNC is GPL-2.0**: unlimited execution, including commercial —
  GPL only restricts distribution. WaaS neither redistributes nor
  modifies the images, so no source obligation attaches. The
  `workspaces-images` Dockerfile repos are MIT (covering only the
  Dockerfiles, not the embedded dependencies): pulling from Docker Hub
  is fine, public republication of built images is to avoid (GPL +
  Ubuntu + app EULAs aggregate). The Kasm Workspaces platform is
  proprietary — **never install a Kasm agent** (what provides
  audio/upload): that is what would fall under their license. The
  standalone server has no platform dependency or phone-home.
